My overall take: this is still moving in the right direction, and bbf5fdf4 is a quality-improving commit. For a merge onto develop, I do not see a new blocker in the latest bundle.

What got better in the latest tip

The strongest new improvement is the MeshInterface wait-state isolation.

Before, the shared wait/error bookkeeping had a real risk of state leakage across sequential waits. The new request-scoped handling is materially better:

wait errors are keyed by (ack_attr, request_id)

stale acknowledgments and stale routing errors are ignored

wait helper cleanup retires request-scoped handlers after completion or timeout

sendData() can clear wait state using the outgoing packet id before send

That is the right fix for the exact class of “one request poisons the next one” bugs that tend to surface in CLI flows and flaky response timing.

The BLE side also improved in sensible ways:

BLE client address is now synchronized back from the underlying Bleak client after connect, disconnect, pair, and unpair

dead weakrefs in the gating registry are pruned before falling back to id(owner) matching

provisional publish-pending state is cleared even when the provisional client was already detached by another path

name-based connects now reserve both the requested alias key and the resolved concrete address key

Those are all good polish passes. They reduce registry drift and narrow race windows rather than expanding them.

The ancillary fixes are also good:

OTA now recomputes firmware size/hash immediately before streaming, which protects against stale metadata

optional powermon backend loading now avoids masking unrelated import failures

smokevirt now validates inline host-port vs explicit port mismatch up front

What I think is solid enough for develop

The BLE connection-management model continues to look stronger than develop:

provisional connect ownership

delayed publication of connected clients

explicit coordination with management operations

better alias/concrete-address reconciliation

improved cleanup around stale or invalidated clients

The latest commit tightened the exact places I would have worried about:

address ownership drift after connect/pair/unpair

dead owner weakrefs leaving false registry state behind

request wait state persisting past timeout boundaries

stale routing responses waking or poisoning a later waiter

That is all good upstream hygiene for an internal merge to develop.

The two remaining things I would still call out
1) Request-scoped waits are better, but same-attribute concurrent waits still are not a first-class supported model

The new logic prevents sequential cross-contamination well. It does not fully generalize to multiple overlapping waits for the same acknowledgment attribute.

Why:

_active_wait_request_ids stores one active request id per acknowledgment attribute

_acknowledgment.receivedTelemetry / receivedPosition / etc. is still a single shared boolean per attribute

starting a new wait for the same attribute supersedes the older one

That is probably fine if the API contract is “only one live telemetry wait at a time,” which seems to be the practical design. But it is worth being explicit. I would not block the merge on this, though I would either document it or add a guard that fails fast if overlapping same-type waits are attempted.

2) close() now swallows TypeError unconditionally in the disconnect-send path

This is the one code-review item that I would keep an eye on.

The new behavior is safer during shutdown noise, but catching TypeError broadly can also hide a genuine programming mistake. In other words, this improves resilience, but slightly weakens bug visibility.

For develop, I would treat this as a minor concern, not a blocker. The simplest refinement would be to keep the suppression but log it a bit more explicitly so real regressions do not disappear silently in debug-only output.
